# GitHub as Single Source of Truth

Dieses Projekt nutzt GitHub ab sofort als **Single Source of Truth** für die operative Projektsteuerung.

## Was bedeutet das?

Operative Informationen werden nicht mehr primär in Markdown-Backlogs oder PM-Dokumenten im Repository gepflegt, sondern in GitHub:
- Issues
- Labels
- Status
- Releases
- Release Notes
- Pull Requests
- Commit- und Tag-Historie

## Was bleibt in den Docs?

Die Dokumentation im Repository bleibt wichtig für:
- Produktvision
- Personas
- User Journey
- Design-Prinzipien
- Architektur
- Domänenmodell
- CI/CD-Zielbild
- Definition of Ready
- Definition of Done
- Branching Strategy
- Getting Started

## Was wird nicht mehr als führend in Docs gepflegt?

Nicht mehr führend im Repo gepflegt werden:
- Backlog-Listen
- operative Roadmap-Details
- Traceability-Matrix als manuelle Tabelle
- PM-Statusmodell als operative Arbeitsliste
- MVP-Issue-Listen als parallele Wahrheit

## Führende operative Quelle

Ab jetzt gilt:
- **GitHub Issues** = Backlog und Umsetzungsstand
- **GitHub Labels** = Status, Priorität, Bereich, Release-Zuordnung
- **GitHub Releases** = veröffentlichte Stände und Release Notes

## Pflegeprinzip

Wenn sich operative Planung ändert, wird sie in GitHub geändert, nicht zuerst in einer Markdown-Datei.
